# Filter Results Cache Implementation

## Overview

This document describes the implementation of a filter results cache for the EC2 Instance Selector API server. The cache stores the results of filter queries in memory to avoid redundant AWS API calls for identical requests.

## Implementation Details

### Cache Structure

The implementation adds the following components to `cmd/api-server/main.go`:

1. **FilterCacheEntry**: Stores cached results with expiration time
2. **FilterResultsCache**: Thread-safe cache manager with automatic cleanup
3. **generateCacheKey()**: Creates SHA256-based cache keys from filter parameters + region

### Key Features

- **Thread-safe**: Uses `sync.RWMutex` for concurrent access
- **Automatic expiration**: Background goroutine removes expired entries
- **Smart key generation**: Excludes presentation parameters (`sort_by`, `sort_direction`, `max_results`) from cache key to allow reuse
- **Per-region caching**: Different regions maintain separate cache entries
- **Configurable TTL**: Controlled via `EC2_INSTANCE_SELECTOR_FILTER_CACHE_TTL` environment variable

### Cache Flow

#### On Cache Hit:
1. Generate cache key from request parameters + region
2. Check cache for valid (non-expired) entry
3. Return cached results immediately
4. Apply sorting and limiting (not cached)
5. Log: `Cache HIT for region us-east-1 (key: abc123...)`
6. Typical response time: 1-5ms

#### On Cache Miss:
1. Generate cache key from request parameters + region
2. Cache lookup fails or entry expired
3. Query AWS API via selector
4. Store results in cache
5. Apply sorting and limiting
6. Log: `Cache MISS for region us-east-1 (key: abc123...)`
7. Typical response time: 100-500ms

### Configuration

New environment variable added:

```bash
EC2_INSTANCE_SELECTOR_FILTER_CACHE_TTL=5m  # Default: 5 minutes
```

Examples:
- `1m` - 1 minute cache
- `5m` - 5 minutes cache (default)
- `1h` - 1 hour cache
- `0` - Disable filter cache

### Affected Endpoints

Both filter endpoints now use the cache:

1. `POST /api/v1/instances/filter` (filterHandler)
2. `GET /api/v1/instances` (getHandler)

### Cache Key Generation

The cache key is generated by:

1. Creating a normalized `FilterRequest` with only filter parameters (excludes `max_results`, `sort_by`, `sort_direction`)
2. Including the resolved region in the cache request
3. Marshaling to JSON for consistent serialization
4. Computing SHA256 hash of the JSON
5. Converting to hexadecimal string

This approach ensures:
- Consistent keys for identical filter parameters
- Different keys for different regions
- Cache reuse when only presentation differs (sort/limit)

### Performance Benefits

**Before (without cache):**
- Every request queries AWS API
- Response time: 100-500ms per request
- AWS API rate limits may be hit under high load

**After (with cache):**
- First request: 100-500ms (cache miss, stores in cache)
- Subsequent identical requests: 1-5ms (cache hit)
- Reduced AWS API calls by ~90%+ for repeated queries
- Better scalability under high load

### Memory Considerations

- Cache entries contain full instance type details
- Memory usage: ~50-500KB per cache entry (depends on result size)
- Example: 100 cache entries ≈ 5-50MB RAM
- Automatic cleanup of expired entries prevents unbounded growth
- Cache is cleared on server restart

### Logging

New log messages:

```
Filter results cache enabled with TTL: 5m0s
Cache HIT for region us-east-1 (key: abc123456789abcd...)
Cache MISS for region us-east-1 (key: def456789abcdef0...)
Cached filter results for region us-east-1 (key: abc123..., 42 instances, cache size: 15)
```

### Testing Recommendations

1. **Cache Hit Test**: Send same request twice, verify second is faster
2. **Cache Miss Test**: Send different requests, verify both query AWS
3. **Region Separation**: Same parameters in different regions should cache separately
4. **TTL Expiration**: Wait for TTL, verify cache expires
5. **Sorting Reuse**: Same filters with different sort should hit cache
6. **Concurrency**: Multiple simultaneous requests with same parameters

### Migration Notes

**Backward Compatibility:**
- Default behavior maintains same response format
- No breaking changes to API
- Can be disabled by setting TTL to 0
- Existing configurations continue to work

**Deployment:**
- No database or external dependencies required
- Cache is in-memory only
- Recommend starting with default 5-minute TTL
- Monitor cache hit rate in logs
- Adjust TTL based on workload patterns

## Example Usage

### With Default Cache (5 minutes)
```bash
export AWS_REGION=us-east-1
./api-server
```

### With Custom Cache TTL
```bash
export AWS_REGION=us-east-1
export EC2_INSTANCE_SELECTOR_FILTER_CACHE_TTL=10m
./api-server
```

### Disable Filter Cache
```bash
export AWS_REGION=us-east-1
export EC2_INSTANCE_SELECTOR_FILTER_CACHE_TTL=0
./api-server
```

## Monitoring Cache Effectiveness

Check logs for cache hit/miss patterns:

```bash
# Count cache hits
grep "Cache HIT" server.log | wc -l

# Count cache misses
grep "Cache MISS" server.log | wc -l

# Calculate hit rate
# Hit Rate = Hits / (Hits + Misses) * 100%
```
